# أكليتو (Akleto) — سياق المشروع الكامل

## عن المشروع
تطبيق أردني لتوصيل مكونات الوجبات (مش وجبات جاهزة). المستخدم يختار وجبات يريد طبخها، التطبيق يحسب المكونات بالكميات الصحيحة عبر خوارزمية دمج ذكي، ويوصّل المكونات من أقرب متجر شريك.

**المستخدمون الأربعة:** زبون، سائق، متجر شريك، أدمن.
**صاحب المشروع:** مؤيد — ليس مبرمجاً، يتوقع من Claude تنفيذ كل الجانب التقني.

## الهوية البصرية
- خلفية: #FDF8F3 كريمي (أبداً أبيض نقي) — للتطبيقات العامة (الزبون، المتجر، السائق)
- عنابي: #A23E48 (اللون الرئيسي)
- خوخي: #F5A65B (ثانوي)
- زيتوني: #65743A (للطزاجة/الصحة فقط — ما تستخدمه لأي شي ثاني)
- ممنوع الزهري أو البرتقالي الفاقع
- لوحة الأدمن تحديداً: باليت غامق مختلف (خلفية #130609، أرقام تتوهج بألوان) — قرار مقصود لأنها لوحة تحكم داخلية، مش واجهة للزبون
- خط Cairo (نص) + Tajawal (عناوين)، RTL كامل
- حواف مدورة 8px+ دائماً

## Firebase
- مشروع: akleto-prod (Firestore + Email/Password Auth + Storage)
- Collections الموجودة:
  - meals: 5 وجبات (منسف، مقلوبة، مسخن، فلافل، فتة) — schema: name_ar, category, difficulty, prep_time_min, base_price, servings_default, description, image_url, is_active, order_count
  - ingredients: 25+ مكوّن — schema: name_ar, unit, category, price_per_unit, image_url, is_active
  - meal_ingredients: سجلات ربط (meal_id, ingredient_id, quantity_per_person) — بنية خوارزمية الدمج
  - stores / orders / drivers / notifications: فاضية، بانتظار بيانات حقيقية

**ملاحظة:** لا يوجد حالياً ملفات akleto-firebase-manager.html أو akleto-firebase-setup.html بالـ repo — إذا احتجناها لازم ننشئها من جديد.

**بنية صفحات الأدمن (مهم لأي صفحة جديدة تنضاف):** كل صفحة أدمن جديدة (مثلاً الطلبات، المتاجر) لازم:
1. تعمل `<link rel="stylesheet" href="admin-style.css">` بدل تكرار الـ CSS
2. تستورد `db, auth, ...` من `./firebase-init.js` بدل تكرار Firebase config
3. تعمل auth gate بسيط: `onAuthStateChanged` يحوّل لـ `akleto-admin.html` لو ما في مستخدم مسجّل دخول
4. تكرر نفس الـ sidebar HTML (بحدد `active` على العنصر المناسب) — لسا مو مستخرج لملف مشترك، فلازم تنسخه يدوياً من أي صفحة أدمن موجودة

## قرارات MVP
- 10-15 وجبة، 2-3 متاجر بعمّان
- ~20 مستخدم للاختبار الداخلي
- مستبعد مبدئياً: تطبيق السائق، الدفع الإلكتروني، برنامج الولاء
- التطبيق: صفحات HTML خام تُنشر عبر GitHub Pages (مش PWA رسمي بعد)

## الأدوات (محدّثة)
- **البناء:** ملفات HTML/JS منفردة (بدون أي أداة No-Code) — **لا نستخدم FlutterFlow إطلاقاً**، هذا قرار نهائي
- **القاعدة الخلفية:** Firebase (Firestore, Auth, Storage)
- **الاستضافة الحالية:** GitHub Pages (مجاني، شغال الآن)
- **GoDaddy:** محجوز للدومين المخصص لاحقاً — مو مستخدم حالياً، رح نفعّله بمرحلة لاحقة بعد الـ MVP
- **VS Code + Live Server:** إلزامي لفتح ملفات Firebase محلياً (بروتوكول file:// ما بيشتغل)
- **Figma:** للتصميم
- ميزانية تسويق مستقبلية: 100-300 دينار/شهر (انستغرام، تيك توك، واتساب)

## آخر نقطة توقفنا (محدّثة)
1. ✅ **صفحة الطلبات بلوحة الأدمن جاهزة** — akleto-admin-orders.html: تعرض كل الطلبات القادمة من تطبيق الزبون **لحظياً** (onSnapshot، مش getDocs، فأي طلب جديد بيظهر فوراً بدون تحديث الصفحة)، فلاتر سريعة بالحالة، الضغط على طلب يفتح تفاصيله (الوجبات المطلوبة + قائمة المكونات المدمجة + السعر التقديري) مع إمكانية تغيير الحالة (pending → preparing → delivering → done / cancelled)
2. **حلقة الاستخدام الأساسية الكاملة صارت شغالة بالكامل تقنياً:** زبون يفتح akleto-customer.html ← يختار وجبات ← يتأكد الطلب ← يظهر فوراً بصفحة الطلبات بالأدمن ← الأدمن يحدّث الحالة
3. القائمة الجانبية بكل صفحات الأدمن الخمسة (admin, orders, meals, ingredients, customers) صارت مربوطة ببعض بروابط حقيقية بدون استثناء

4. ✅ **صفحة الطلبات صارت متجاوبة (responsive) بشكل كامل بين الموبايل والكمبيوتر**: أُضيفت حاوية `.orders-content-max` (max-width: 1400px, margin: auto) حول الفلاتر وقائمة البطاقات عشان ما تتمدد البطاقات بشكل مبالغ فيه على شاشات الكمبيوتر العريضة، مع breakpoint إضافي بين 601px-900px (تابلت) يفرض عمودين ثابتين بدل الاعتماد فقط على auto-fill. شبكة البطاقات على الموبايل (تحت 480px) تبقى عمود واحد كما كانت.

5. ✅ **إصلاح باغ حقيقي مؤكد بالصورة**: على الموبايل، عند فتح القائمة الجانبية (sidebar) كانت طبقة التعتيم الغامقة خلفها (`.sidebar-overlay`) ما بتظهر، فتضل الصفحة اللي تحتها ظاهرة وقابلة للتفاعل جنب القائمة. السبب: خاصية CSS `inset: 0` (المستخدمة بـ 4 عناصر: sidebar-overlay, modal-overlay, fullscreen-page, login-gate بملف admin-style.css + عنصر مشابه بـ akleto-customer.html) مش مدعومة على بعض متصفحات أندرويد الأقدم/الأوفر، فالعنصر ما ياخد حجم الشاشة كاملة. تم استبدالها بـ `top:0; left:0; right:0; bottom:0;` الصريحة بكل المواضع الخمسة — إصلاح بملف CSS مشترك واحد بينطبق تلقائياً على كل صفحات الأدمن.

6. ✅ **إصلاح مشكلة "الحجم كبير وغير مناسب" بصفحة الطلبات على الموبايل**: السبب المرجّح هو خاصية معروفة بمتصفحات أندرويد الأقدم اسمها **font boosting** — بتكبّر النص تلقائياً بشكل غير متوقع على الشاشات الضيقة بغض النظر عن أحجام الخط المحددة بالـ CSS. تم تعطيلها بإضافة `text-size-adjust: 100%` على `html` بكل صفحات الأدمن (admin-style.css) وتطبيق الزبون. بالإضافة: أضفت `overflow-x: hidden` على html/body لمنع أي تمدد أفقي غير مقصود، وأضفت رقم إصدار `?v=3` لرابط admin-style.css بكل صفحات الأدمن لضمان تحميل النسخة المحدثة دائماً بدل نسخة قديمة من كاش المتصفح.

7. 🗑️ **تم حذف ملف `akleto-admin-orders.html` نهائياً** (بناءً على طلب صريح مؤكَّد). ⚠️ رابط "الطلبات" لسا موجود بالقائمة الجانبية بكل صفحات الأدمن الباقية (`akleto-admin.html`, `akleto-admin-meals.html`, `akleto-admin-ingredients.html`, `akleto-admin-customers.html`) وبيأدي لصفحة 404 لأنه ما انحذف عمداً بناءً على طلب المستخدم. بيانات الطلبات نفسها ما زالت موجودة بـ Firestore (collection `orders`) — بس ما في واجهة حالياً لعرضها أو إدارتها.

# أكليتو (Akleto) — سياق المشروع الكامل

## عن المشروع
تطبيق أردني لتوصيل مكونات الوجبات (مش وجبات جاهزة). المستخدم يختار وجبات يريد طبخها، التطبيق يحسب المكونات بالكميات الصحيحة عبر خوارزمية دمج ذكي، ويوصّل المكونات من أقرب متجر شريك.

**المستخدمون الأربعة:** زبون، سائق، متجر شريك، أدمن.
**صاحب المشروع:** مؤيد — ليس مبرمجاً، يتوقع من Claude تنفيذ كل الجانب التقني.

## الهوية البصرية
- خلفية: #FDF8F3 كريمي (أبداً أبيض نقي) — للتطبيقات العامة (الزبون، المتجر، السائق)
- عنابي: #A23E48 (اللون الرئيسي)
- خوخي: #F5A65B (ثانوي)
- زيتوني: #65743A (للطزاجة/الصحة فقط — ما تستخدمه لأي شي ثاني)
- ممنوع الزهري أو البرتقالي الفاقع
- لوحة الأدمن تحديداً: باليت غامق مختلف (خلفية #130609، أرقام تتوهج بألوان) — قرار مقصود لأنها لوحة تحكم داخلية، مش واجهة للزبون
- خط Cairo (نص) + Tajawal (عناوين)، RTL كامل
- حواف مدورة 8px+ دائماً

## Firebase
- مشروع: akleto-prod (Firestore + Email/Password Auth + Storage)
- Collections الموجودة:
  - meals: 5 وجبات (منسف، مقلوبة، مسخن، فلافل، فتة) — schema: name_ar, category, difficulty, prep_time_min, base_price, servings_default, description, image_url, is_active, order_count
  - ingredients: 25+ مكوّن — schema: name_ar, unit, category, price_per_unit, image_url, is_active
  - meal_ingredients: سجلات ربط (meal_id, ingredient_id, quantity_per_person) — بنية خوارزمية الدمج
  - stores / orders / drivers / notifications: فاضية، بانتظار بيانات حقيقية

**بنية صفحات الأدمن (مهم لأي صفحة جديدة تنضاف):** كل صفحة أدمن جديدة (مثلاً الطلبات، المتاجر) لازم:
1. تعمل `<link rel="stylesheet" href="admin-style.css">` بدل تكرار الـ CSS
2. تستورد `db, auth, ...` من `./firebase-init.js` بدل تكرار Firebase config
3. تعمل auth gate بسيط: `onAuthStateChanged` يحوّل لـ `akleto-admin.html` لو ما في مستخدم مسجّل دخول
4. تكرر نفس الـ sidebar HTML (بحدد `active` على العنصر المناسب) — لسا مو مستخرج لملف مشترك، فلازم تنسخه يدوياً من أي صفحة أدمن موجودة
5. ⚠️ **`akleto-admin.html` فيه نسخة مكرّرة كاملة من نموذجي "إضافة وجبة" و"إضافة مكوّن"** (منطق مستقل عن `akleto-admin-meals.html`/`akleto-admin-ingredients.html`، مش استدعاء مشترك) — أي تغيير بمخطط بيانات المكونات أو الوجبات (حقول جديدة، تغيير أسماء حقول...) **لازم يتكرر يدوياً بثلاث أماكن**: الصفحة المستقلة + `akleto-admin.html`. نسيان مزامنة `akleto-admin.html` بيخلق تضارب بيانات صامت (نموذج بيكتب مخطط قديم بينما باقي النظام يتوقع الجديد).

15. 🆕 **إعادة تصميم نظام المقادير من الصفر: "نوع القياس + العبوة الحقيقية + وحدة الوصفة المطبخية"** (بديل نهائي — تجاوز محاولتين سابقتين أُلغيتا: أولاً حقل `min_purchase_qty` بسيط، وثانياً تصنيف الوحدات لـ"مغلّفة/سائبة". القرار النهائي بعد نقاش مباشر مع مؤيد لأن الحلول السابقة عقّدت شاشة إدخال المكوّن أو ما عالجت حالة "رشة ملح من كيس 250غ" بشكل صحيح):

   **المشكلة الأصلية:** بعض المكونات تُستخدم بالوصفة بكمية صغيرة جداً (رشة ملح) بينما أصغر وحدة مبيعة بالسوق أكبر بكثير (كيس ملح 250غ). محتاجين ندخل الوجبة بوحدة مطبخية طبيعية (رشة) وفي نفس الوقت نعرف نحسب كم عبوة سوق فعلياً لازم نشتري.

   **الحل المعتمد — 3 مفاهيم منفصلة:**
   - **نوع القياس** (`measure_type`): `weight` (وزن) / `volume` (حجم) / `count` (عدد) — خاصية على كل مكوّن بشاشة المكونات.
   - **العبوة الحقيقية** (كيف يُشترى المكوّن فعلياً بالسوق): `package_label` (اسم حر: "كيس"، "علبة"، "كرتونة"، "حبة") + `package_qty` (الكمية بوحدة الأساس — غرام لو وزن، مل لو حجم، عدد لو عدد) + `price_per_package` (سعر العبوة الكاملة).
   - **وحدة الوصفة** (كيف تُدخل الكمية بشاشة الوجبة): قائمة وحدات مطبخية جاهزة حسب نوع القياس (رشة/معلقة صغيرة/معلقة كبيرة/غرام/كيلو للوزن، معلقة صغيرة/معلقة كبيرة/مل/لتر/كأس صغير/كأس كبير للحجم، حبة للعدد) — كل وحدة إلها عامل تحويل ثابت لوحدة الأساس.

   **`firebase-init.js` — مصدر الحقيقة المشترك الجديد:**
   - `export const MEASURE_TYPES` = كائن يحدد لكل نوع قياس: `label`, `baseUnitLabel`, و`recipeUnits` (خريطة اسم الوحدة ← عامل التحويل لوحدة الأساس). مثال: `weight.recipeUnits = {'رشة':1, 'معلقة صغيرة':5, 'معلقة كبيرة':15, 'غرام':1, 'كيلو':1000}`.
   - `export function convertToBase(measureType, recipeUnit, qty)` — يحوّل كمية بوحدة وصفة لكمية بوحدة الأساس. **مستوردة بكل من `akleto-admin-ingredients.html`, `akleto-admin-meals.html`, `akleto-customer.html`, و`akleto-admin.html`** لضمان عدم اختلاف جدول التحويل بين الصفحات.

   **مخطط بيانات `ingredients` الجديد** (استُبدل بالكامل، القديم `unit`/`price_per_unit` أُزيل): `measure_type`, `package_label`, `package_qty`, `price_per_package`.

   **مخطط بيانات `meal_ingredients` الجديد** (استُبدل `quantity_per_person` بحقلين): `recipe_unit` (اسم وحدة الوصفة المختارة، مثل "رشة") و `recipe_qty` (الكمية بهاي الوحدة، مثل 1).

   **شاشة المكوّن (`akleto-admin-ingredients.html` و نسخة `akleto-admin.html` المطابقة):** نموذج إضافة مكوّن صار: الاسم → نوع القياس (select) → التصنيف → اسم العبوة (نص حر) → كمية العبوة (label بتتغيّر تلقائياً حسب نوع القياس: غرام/مل/حبة) → سعر العبوة كاملة → صورة.

   **شاشة الوجبة (`akleto-admin-meals.html` و نسخة `akleto-admin.html` المطابقة):** كل صف مكوّن فيه: حقل بحث اسم المكوّن (datalist كالسابق) → حقل كمية رقمي → **select وحدة وصفة** يتحدّث تلقائياً حسب نوع قياس المكوّن المختار (بدون إعادة رسم كامل الصف — بس تحديث الـ select بالـ DOM مباشرة عشان ما نخسر تركيز حقل الكتابة). مثال إدخال: "ملح" + "1" + "رشة".

   **`akleto-customer.html` — خوارزمية الدمج الجديدة (`computeMergedIngredients`):**
   1. لكل وجبة مختارة، `getMealIngredients` تجيب `recipe_unit`+`recipe_qty` من الربط، وتحوّلها لـ`baseQtyPerPerson` عبر `convertToBase`.
   2. تُجمع `baseQtyPerPerson × عدد الأشخاص` لكل مكوّن عبر كل الوجبات بالطلب (بوحدة الأساس: غرام/مل/حبة).
   3. `purchaseQty = Math.ceil(baseQty_الإجمالي / package_qty)` — عدد العبوات الكاملة الواجب شراؤها (ما يمكن شراء نص عبوة).
   4. شاشة السلة تعرض `purchaseQty × package_label` (مثال: "1 كيس")، ولو الكمية المشتراة أكبر من المطلوب فعلياً تظهر ملاحظة توضيحية: "الوصفة تحتاج تقريباً X غرام — رح تشتري Y كيس (Z غرام)".
   5. **السعر التقديري = `purchaseQty × price_per_package`** (سعر ما يُشترى فعلياً، مش كسر نظري).

   **بنية الطلب بـ Firestore:** كل عنصر بـ `merged_ingredients` صار: `ingredient_id`, `name`, `package_label`, `purchase_quantity` (عدد العبوات)، `base_qty_needed` (الكمية الفعلية اللي تحتاجها الوصفة بوحدة الأساس، للمرجعية)، `base_unit_label`.

   **`akleto-admin-orders.html`:** تفاصيل الطلب تعرض `purchase_quantity × package_label` كرقم رئيسي (هذا يلي لازم يُشترى)، مع ملاحظة صغيرة تحته توضح الكمية الفعلية اللي تحتاجها الوصفة — مفيدة للمتجر عند التسوق الفعلي.

   **⚠️ ملاحظة حاسمة — بدء نظافة:** مؤيد أكّد إنه نبلش نظافة تماماً (لا يوجد بيانات إنتاجية حالياً تستاهل ترحيل). **أي مكوّن أو وجبة قديمة بقاعدة البيانات من قبل هاد التغيير أصبحت غير متوافقة مع الشاشات الجديدة ولازم تُدخل من جديد بالكامل** (نوع القياس، العبوة، ووحدة كل مقدار بكل وجبة) — الحقول القديمة (`unit`, `price_per_unit`, `quantity_per_person`) ما عاد أي شاشة تقرأها أو تكتبها.

16. 🆕 **تعديل المكونات بالضغط عليها (`akleto-admin-ingredients.html`):** كل صف بقائمة المكونات صار قابل للضغط (`cursor:pointer`) ويفتح نفس مودال الإضافة لكن بوضع تعديل (`currentEditingIngredientId`) — العنوان يصير "✏️ تعديل: [الاسم]"، الحقول تتعبى من بيانات المكوّن الحالي، وزر "🗑️ حذف المكوّن" يظهر بأسفل المودال. الحفظ يستخدم `updateDoc` بدل `addDoc`. زر الحذف "✕" على الصف نفسه لسا شغال بشكل منفصل (`event.stopPropagation()` يمنعه من فتح مودال التعديل بالغلط).

17. 🆕 **سلة تطبيق الزبون صارت قابلة للتعديل يدوياً (`akleto-customer.html`):**
   - كل سطر بقائمة "المكونات المدمجة" بالسلة صار يعرض: سعر العبوة الواحدة، stepper (−/+) لزيادة/إنقاص عدد العبوات، السعر الإجمالي للسطر، وزر ✕ لحذف المكوّن بالكامل من السلة.

18. 🆕 **إدارة توفر المنتجات بالمتجر (\"منتجاتي\") + توزيع الطلبات تلقائياً على أقرب متجر متوفر فيه كل المكونات:**

   **الفكرة:** كل متجر شريك صار يحدد بنفسه (من تطبيق الزبون بوضع المتجر) أي المكونات متوفرة عنده فعلياً وبأي سعر، وعملية توزيع الطلب على متجر تصير تلقائية بالكامل بناءً على: (أ) توفر **كل** مكونات الطلب عند المتجر، و(ب) الأقرب جغرافياً للزبون — **الشرطان لازم يتحققا معاً**، الأولوية ليست فقط للمسافة.

   **مجموعة Firestore جديدة: `store_ingredients`** — معرّف الوثيقة `{storeId}_{ingredientId}` (يمنع تكرار)، الحقول: `store_id`, `ingredient_id`, `is_available` (bool), `price_per_package` (سعر المتجر الخاص، قد يختلف عن سعر المكوّن العام), `updated_at`.
   - **عدم وجود وثيقة** لمكوّن معيّن عند متجر = المكوّن **\"جديد\"** (لم يراجعه المتجر بعد).

   **`akleto-customer.html` — شاشة \"منتجاتي\" بوضع المتجر (`storeProductsScreen`):**

   **3 مانيفستات منفصلة (كل تطبيق يُثبّت بشكل مستقل بأيقونة واسم خاص فيه):**
   - `manifest-customer.json` — start_url `akleto-customer.html`، ثيم عنابي #A23E48 على خلفية كريمي #FDF8F3.
   - `manifest-driver.json` — start_url `akleto-driver.html`، نفس هوية الزبون البصرية.
   - `manifest-admin.json` — start_url `akleto-admin.html`، ثيم غامق #130609 (يطابق هوية لوحة الأدمن المنفصلة).

   **أيقونات مُصمَّمة حسب الدور** (`icons/` — مقاسات 192/512/180 لكل تطبيق، مولّدة برمجياً):
   - الزبون: سلة تسوق بيضاء على خلفية عنابي.
   - السائق: مقود سيارة أبيض على خلفية عنابي (نفس اللون، شكل مختلف للتمييز).
   - الأدمن: ترس (gear) خوخي على خلفية غامقة.

   **`sw.js` — Service Worker مشترك** (يُسجَّل من الثلاث تطبيقات): استراتيجية network-first (يجيب النسخة الأحدث دايماً أثناء التطوير المستمر) مع fallback للكاش لو الاتصال انقطع لحظياً. بيتجاهل أي طلب مش GET أو مش من نفس الأصل (يعني ما يتدخل بطلبات Firestore/Google APIs إطلاقاً).

   **كل صفحة (10 ملفات: الزبون، السائق، وكل صفحات الأدمن الثمانية) صار فيها:**
   - `<link rel="manifest">` يشاور على المانيفست المناسب لدورها
   - `<meta name="theme-color">` مطابق لهوية كل تطبيق
   - وسوم iOS (`apple-mobile-web-app-capable`, `apple-touch-icon`, إلخ) لدعم "أضف للشاشة الرئيسية" على آيفون كمان
   - سكريبت تسجيل الـ Service Worker بنهاية `<head>`

   **⚠️ ملاحظة مهمة:** هاي خطوة تثبيت (installability) بس — **مش بناء APK حقيقي**. لو احتجنا لاحقاً توزيع Play Store رسمي أو APK مستقل، الخطوة الجاية هي Capacitor (يحتاج Android Studio/SDK محلياً عند مؤيد، ما ينبنى من هون).

   **للتجربة:** بعد ما يوصل التحديث لـ GitHub Pages (1-2 دقيقة)، افتح كل رابط من متصفح Chrome على أندرويد → قائمة المتصفح (⋮) → "إضافة إلى الشاشة الرئيسية" / أو بيظهر بانر تلقائي بالأسفل. على آيفون: زر المشاركة → "إضافة إلى الشاشة الرئيسية".

23. 🆕 **شاشة الإشعارات بلوحة التحكم (`akleto-admin-notifications.html`) — الجزء 1 من 3 (استقبال + إرسال):**

   ربط التنبيهات **التلقائية** بمجموعة `admin_notifications` (تنبيه أوتوماتيكي للأدمن عند طلب جديد / طلب جاهز / طلب موصّل) — لسا ما تربط، فـ"الوارد" بلوحة التحكم يضل فاضي إلا لو حدا كتب فيه يدوياً.

25. 🆕 **ربط التنبيهات التلقائية بالأدمن (الجزء 3 من 3 — اكتمل نظام الإشعارات كاملاً):**

   4 أحداث بدورة حياة الطلب تكتب تلقائياً بمجموعة `admin_notifications` (كل وحدة `read:false` عند الإنشاء):
   - **`new_order`** — بتطبيق الزبون، فور `placeOrder()` تنجح، مباشرة بعد ما يتحدد المتجر.
   - **`order_ready`** — بوضع المتجر (`akleto-customer.html`)، فور `markStoreOrderReady()`.
   - **`order_claimed`** 🆕 (نوع جديد ما كان بالتصميم الأولي، أضفناه لتغطية دورة الحياة كاملة) — بتطبيق السائق، فور نجاح `claimOrder()` (Transaction الاستلام).
   - **`order_delivered`** — بتطبيق السائق، فور `markDelivered()`.

   كل تنبيه فيه `title` (يتضمن كود الطلب بصيغة `#XXXX` عبر دالة `ticketCode()` الموجودة أصلاً بكل تطبيق) و`message` (بيذكر اسم الزبون/السائق/المتجر المعنيين) و`related_id` (معرّف الطلب، لاستخدام مستقبلي لو حبينا نربط النقر بالتنبيه لفتح تفاصيل الطلب مباشرة).

   **ملاحظة موثوقية:** كتابة `admin_notifications` ما بتوقف أو تفشّل العملية الأساسية لو صار فيها خطأ — مكتوبة بـ `.catch()` منفصل (fire-and-forget) بعد نجاح تحديث/إنشاء الطلب نفسه، عشان أي مشكلة بكتابة التنبيه ما تأثر على تجربة المستخدم الأساسية (تأكيد الطلب، تحديث الحالة...).

   **✅ نظام الإشعارات الآن مكتمل بالكامل بأجزائه الثلاثة:** شاشة أدمن (وارد+إرسال) ← استقبال فعلي (زبون/سائق/متجر) ← تنبيهات تلقائية بدورة حياة الطلب.

   - كل عنصر تنقّل بأيقونة داخل مربع ملوّن (`ic-box`) بدل إيموجي عادي + سهم (`chev`).
   - إصلاح جانبي: `.cs-overlay` كان يستخدم `inset:0` (نفس الباغ المعروف بأندرويد القديم) — استُبدل بـ `top/left/right/bottom` صريحة.

   - 🆕 **أُضيفت 3 عناصر جديدة تحت قسم "الحساب":**
     1. **📍 عنواني** — شاشة كاملة جديدة (`addressScreen`)، **عنوان محفوظ فعلياً** بوثيقة الزبون (`customers/{customerId}`): حقول `address_area` (المنطقة/الحي)، `address_details` (تفاصيل الشارع/البناية)، `address_notes` (ملاحظات اختيارية)، `address_updated_at`. يُحمَّل عبر `getDoc` (استُورد حديثاً من `firebase-init.js` — كان موجود بالتصدير أصلاً بس غير مستخدم بهاد الملف) عند فتح الشاشة، ويُحفظ عبر `updateDoc`. **⚠️ ملاحظة مهمة:** هاد العنوان محفوظ للمرجعية فقط حالياً — **لسا ما انربط بمنطق "أقرب متجر"** (`findBestStoreForIngredients`) اللي لسا يعتمد كلياً على `navigator.geolocation` الحي وقت الطلب (نقطة 18 بالتاريخ القديم). ربطه بحساب المسافة خطوة مستقبلية منفصلة.

     3. **⚙️ الإعدادات** — شيت "قريباً" فاضي مبدئياً (بناءً على تأكيد مؤيد).
   - شيت "قريباً" مشترك (`comingSoonSheet` / `COMING_SOON_CONTENT` بالـ JS) — قابل لإعادة الاستخدام لأي ميزة مستقبلية بس نضيف مفتاح جديد بـ `COMING_SOON_CONTENT` بدل بناء شيت جديد من الصفر.

   **ج) شاشة تفاصيل الوجبة (bottom sheet):** توضيح مهم بناءً على طلب مؤيد — كل مكوّن صار يعرض سطرين منفصلين: **"يحتاج: X {وحدة الوصفة}"** (كمية الوصفة بالضبط) و **"🛍️ رح تشتري: Y {اسم العبوة}"** (الكمية الفعلية اللي رح تُشترى بعد التقريب لعبوات كاملة) — لتفادي التباس الزبون (مثال حقيقي: "خبز برغر" وصفته تحتاج 2 حبة بس بيتباع بعبوة 6 حبات، فكان يخاف يفهم غلط إنه رح يشتري 2 كيس). **لا تضيف أي جملة شرح عامة/بانر توضيحي فوق القائمة** — تمت تجربتها وطُلب حذفها صراحة لأنها غير مفهومة/زايدة عن الحاجة؛ الوضوح المطلوب هو بس بالتسمية المباشرة "يحتاج" مقابل "رح تشتري" بدون أي نص إضافي.

27. 🆕 **شاشة تأكيد الطلب الجديدة (بين السلة وإرسال الطلب فعلياً):**

   **الفكرة:** زر "تأكيد الطلب" بشاشة السلة ما عاد يرسل الطلب مباشرة — صار يفتح شاشة مراجعة جديدة (`orderConfirmScreen`)، وهاي الشاشة فيها زر "تأكيد الطلب" الحقيقي اللي يستدعي `finalizeOrder()` (كانت اسمها `placeOrder()` وانحذفت بالكامل، استُبدلت).

   **محتوى الشاشة الجديدة:**
   - عناصر الطلب (اسم الوجبة + عدد الأشخاص) + زر "عرض التفاصيل الكاملة" يفتح قائمة المكونات المدمجة الكاملة (نفس بيانات `getFinalCartIngredients()` المستخدمة بالسلة).
   - تفاصيل التوصيل: اسم المستلم ورقم الهاتف (من الجلسة) + **عنوان التوصيل** (يُجلب حياً بـ `getDoc` من وثيقة الزبون `customers/{customerId}` كل مرة تُفتح الشاشة — مو من كاش). **لو ما في عنوان محفوظ (`address_area` فاضي)، يظهر تحذير واضح مع رابط "أضف عنوانك الآن" ويمنع تأكيد الطلب** (فحص إلزامي داخل `finalizeOrder` كمان، مو بس بالواجهة).
   - حقل **"ملاحظات إضافية للتاجر أو السائق"** (اختياري، `textarea` جديد `#oc-notes`) — يُحفظ بحقل `customer_notes` على وثيقة الطلب، ويظهر لكل من المتجر (بشيت طلبات المتجر بنفس ملف الزبون) والسائق (`akleto-driver.html`).
   - طريقة الدفع: صف ثابت غير تفاعلي "الدفع عند الاستلام 💵" (بما إنه الدفع الإلكتروني لسا مستبعد من الـ MVP — لا يوجد خيار بطاقة).
   - ملخص الفاتورة: إجمالي الطلب (subtotal من المكونات) + **رسوم التوصيل (ثابتة 1.00 د.أ — ثابت `DELIVERY_FEE`)** + **"بدل الخدمة" (⚠️ مش "الضريبة" — تسمية متعمّدة بناءً على طلب مؤيد)** = 4% من قيمة الطلب قبل أي خصومات، بحد أدنى 0.3 د.أ وحد أعلى 1.0 د.أ (`computeServiceFee()` بمنطق `Math.min(1.0, Math.max(0.3, subtotal*0.04))`) + المجموع النهائي.

   **حقول جديدة على وثيقة `orders`:**
   - `delivery_address` (نص مدمج: المنطقة + التفاصيل + ملاحظة العنوان بين قوسين إن وُجدت)
   - `customer_notes` (ملاحظة الطلب نفسها، منفصلة عن ملاحظة العنوان المحفوظة بحساب الزبون)
   - `subtotal_price`, `delivery_fee`, `service_fee` (تفصيل الفاتورة، بالإضافة لـ `total_estimated_price` الموجود سابقاً واللي صار الآن = subtotal + delivery_fee + service_fee)
   - `payment_method` (حالياً دايماً `'cash_on_delivery'` — مجرد قيمة ثابتة تمهيداً لدعم طرق دفع تانية بالمستقبل)

   **تطبيق السائق (`akleto-driver.html`):** أُضيف صف "📍 عنوان التوصيل" (`sheet-address`) وصندوق ملاحظة الزبون (`sheet-notes-wrap`) بشيت تفاصيل الطلب — **هاي أول مرة السائق يشوف فيها عنوان توصيل فعلي** (كان الفجوة المذكورة بالنقطة "على الأفق" بالتاريخ القديم؛ الحل هلق هو نص جاهز محفوظ بحساب الزبون، مش خوارزمية توجيه GPS حية).

   **شاشة طلبات المتجر (بنفس `akleto-customer.html`، وضع المتجر):** أُضيف عرض ملاحظة الزبون (`so-notes-wrap`) بشيت تفاصيل الطلب الوارد، لو موجودة.
